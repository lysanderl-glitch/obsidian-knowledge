## 一、核心架构

```
Harness Error (Webhook)
       ↓
   n8n Workflow Engine
       ↓
┌─────────────────────────────┐
│  Python 分析服务器           │
│  (harness-analyze-server)   │
│                             │
│  - 调用 Claude API (MiniMax) │
│  - 解析 thinking/text 结构   │
│  - 返回结构化 JSON           │
└─────────────────────────────┘
       ↓
  置信度判定
       │
  ┌────┴────┐
  │ 高置信度 │ 低置信度
  ↓         ↓
自动执行   Slack通知
  ↓         ↓（人工介入后记录结果）
验证结果     ↓
  │      回流到AI Loop
  ↓
┌─────────────────────────────┐
│  Learning Layer             │
│  (上下文累积 + 记忆更新)     │
└─────────────────────────────┘
```

**技术选型说明：**
- AI Provider: Claude API via MiniMax 代理（`https://api.minimaxi.com/anthropic`）
- 分析服务器：Python HTTP Server（独立进程，不在 Docker 内）
- n8n：仅做路由和通知，不直接调用 AI（避免 Docker 环境限制）

## 二、AI Agent Loop 详细流程

### Step 1: Think (分析)
- **输入**：错误信息、错误类别、上下文历史
- **检索**：错误库记忆 + 知识库 + 历史解决方案
- **判断**：已知问题？未知问题？复杂问题？
- **输出**：修复方案 + 置信度 + 预估时间

### Step 2: Act (执行)
- **高置信度(≥90%)** → 自动执行修复
- **中置信度(50-79%)** → 执行 + 密切监控
- **低置信度(<50%)** → 通知人工 + 等待

### Step 3: Observe (观察)
- 监控执行结果（1min / 5min / 15min）
- 判断：是否解决了根因？
- 输出：成功 / 失败 / 部分成功

### Step 4: Correct (纠正)
- 失败 → 简化任务 + 重新 Think
- 连续失败3次 → 停止 + 通知
- 成功 → 记录到记忆 + 闭环

## 三、置信度与行动矩阵

| 置信度 | 条件 | 行动 | 人工介入 |
|--------|------|------|----------|
| **≥90%** | 错误明确、修复方案确定、历史成功率高 | 自动执行 + 自动验证 | 无 |
| **70-89%** | 错误明确、修复方案有一定不确定性 | 执行 + 5分钟监控 | 失败则介入 |
| **50-69%** | 错误较明确但修复方案有风险 | 执行 + 实时监控 | 全程可见 |
| **<50%** | 错误复杂或未知 | 通知人工 + 附分析 | 必须介入 |

## 四、风险 Register

| 风险 | 级别 | 描述 | 缓解策略 | 可接受程度 |
|------|------|------|----------|------------|
| **R1: AI 误判导致错误修复** | 高 | 执行了错误操作导致更大问题 | 高风险操作（删除/回滚）强制人工确认；保留操作前快照 | 可接受（有兜底） |
| **R2: AI 无限循环尝试** | 高 | 反复失败但持续重试 | 最多3次循环后强制停止 + 通知 | 可接受（有上限） |
| **R3: 升级过度（鸡毛蒜皮也通知）** | 中 | 低级错误也骚扰人工 | 升级阈值动态调整；用户反馈调整置信度 | 可接受（可 tune） |
| **R4: AI 学会错误模式** | 中 | 反馈数据被污染 | 人工审核通过的修复才能进入记忆 | 可接受（质量门禁） |
| **R5: 成本失控** | 中 | 频繁调用 Gemini 导致费用暴涨 | 月度调用上限告警；按错误复杂度分流 | 可接受（可预算控制） |
| **R6: 执行超时/系统故障** | 低 | n8n 或 Gemini 不可用 | 降级：超时自动通知人工；定期健康检查 | 可接受（有降级方案） |
| **R7: Slack 通知疲劳** | 低 | 大量通知导致忽略真正问题 | 通知分级 + 汇总报告 | 可接受（可 tune） |

## 五、实施阶段

### Phase 1: 基础闭环（1-2周）

**目标**：跑通最小的 self-healing loop

```
Harness Error → Webhook → n8n → Gemini 分析 → Slack 通知
                                              ↓
                                        AI 给出建议
                                              ↓
                                        人工执行 + 记录结果
```

- 只做 **分析和建议**，不自动执行
- 积累第一批真实错误数据
- 验证 AI 分析质量

**里程碑**：连续7天 AI 建议质量达标

**人工参与点**：
- 执行 AI 给出的修复建议
- 记录修复结果（成功/失败）

---

### Phase 2: 受控自动执行（2-3周）

**目标**：高置信度错误自动执行

```
Harness Error → Webhook → n8n → Gemini → 置信度≥90% → 自动执行
                                        ↓
                                  验证结果 → 记录学习
```

- 只对 **高置信度(≥90%)** 错误自动执行
- 执行范围：**重启服务/重试构建/清理缓存/通知相关人**
- 不涉及：**删除/回滚/配置变更**

**里程碑**：自动执行成功率 ≥ 85%

**人工参与点**：
- 审核 AI 的自动执行结果（每日）
- 处理失败的自动执行

---

### Phase 3: 智能扩展（持续）

**目标**：逐步降低置信度阈值 + 扩大执行范围

```
Phase 2 成功 → 置信度阈值降至 80%
Phase 3 成功 → 阈值降至 70% + 扩大执行范围
...
最终 → 置信度≥50% 即可自动处理大多数错误
```

**持续任务**：
- 每月复盘：自动执行的错误，成功率多少？
- 每月学习：从失败案例优化 AI 提示词
- 每季度：扩大执行范围评估

**人工参与点**：
- 月度复盘会议
- 季度执行范围评估决策

## 七、验证与学习机制

### 验证（Verification）
```
自动执行后：
1. 检查 Harness 状态（是否变 green）
2. 等待观察窗口（1min / 5min / 15min）
3. 成功 → 记录到 Success Log
4. 失败 → 记录到 Failure Log + 通知
```

### 学习（Learning）
```
每次处理结果回流：
1. 成功的修复 → 追加到上下文（未来同类型更快处理）
2. 失败的尝试 → 记录失败原因 + 下次不重蹈覆辙
3. 人工介入的案例 → 人工决策记录 + AI 学习
```

### 记忆更新
```
每个错误处理闭环后：
→ 更新错误库（pattern + 解决方案）
→ 更新置信度（基于历史成功率调整）
→ 更新提示词（从失败案例优化）
```

## 八、方案乙 vs 方案甲对比

| 维度 | 方案甲（规则驱动） | 方案乙（AI 驱动） |
|------|--------|--------|
| **错误覆盖** | 只处理已知 catalog | 处理已知 + 未知 |
| **维护成本** | 高（catalog 需持续扩充） | 低（AI 自动学习） |
| **人工干预** | 多（catalog 不全就需介入） | 少（随 AI 能力增长递减） |
| **初期投入** | 高（catalog 设计 + 全覆盖） | 中（搭建 loop + 调优） |
| **长期价值** | 递减（维护成本持续） | 递增（越用越聪明） |
| **风险曲线** | 初期低，长期因维护不足升高 | 初期中高，随学习降低 |

## 十、调试记录与Lessons Learned

### 2026-04-07 Phase 1 部署调试

**问题1：分析服务器返回空 analysis**

- **现象**：`curl` 调用超时，n8n workflow 无法获得分析结果
- **根因**：MiniMax-M2 模型返回 `content` 数组，第一个元素是 `type: "thinking"`，第二个才是 `type: "text"`
- **解决**：在 API payload 中添加 `"thinking": {"type": "disabled"}`
- **代码**：
  ```python
  api_payload = {
      "model": "claude-3-5-sonnet-20241022",
      "max_tokens": 500,
      "thinking": {"type": "disabled"},  # 关键修复
      "messages": [{"role": "user", "content": prompt}]
  }
  ```

**问题2：端口占用（OSError: [Errno 98] Address already in use）**

- **解决**：使用 `socketserver.TCPServer.allow_reuse_address = True` 或换端口
- **最终方案**：端口从 5800 改为 5810

**问题3：JSON 响应被截断**

- **现象**：`confidence` 和 `category_confirmed` 返回 `null`
- **根因**：`max_tokens=300` 不足以返回完整 JSON
- **解决**：增加到 `max_tokens=500`

**问题4：n8n Docker 环境限制**

- **现象**：Code 节点无法写入宿主机文件系统
- **解决**：使用 Python 服务器处理文件 IO，n8n 仅做路由

### 技术要点总结

| 要点 | 值 |
|------|-----|
| n8n 容器 → 宿主机 | 使用 `172.18.0.1` 而非 `127.0.0.1` |
| MiniMax API 响应格式 | content 数组需遍历找 `type: "text"` |
| AI thinking 模式 | 需显式禁用才能获取纯净文本响应 |
