阶段二

项目级协作

预计 3-5 天

核心理念：用 CLAUDE.md 定义项目规范，让 AI 持续输出一致风格的代码。

学习内容

  • CLAUDE.md 编写：定义技术栈、代码风格、架构决策、常用命令
  • 版本控制纪律：每个功能一个分支，AI 生成的代码必须 review
  • Specification Engineering：写清楚需求规格比写 prompt 更重要
  • 测试驱动：让 Claude Code 生成代码的同时生成测试

关键技巧

CLAUDE.md 就是你的"项目宪法"，写得越清楚 AI 输出越靠谱。
用 Plan Mode 先设计方案再写代码，避免返工。
每次改动后跑测试，AI 生成的代码约有 45% 存在潜在问题。

避坑指南

常见陷阱与解决方案

1. Prompt 太模糊
"帮我做个网站" → "用 Next.js + Tailwind 做一个展示日记的单页应用，支持按日期筛选"
2. 没有 CLAUDE.md
AI 每次都在猜你的偏好，输出不稳定。写好项目规范文件是第一步。
3. 一次塞太多
一个 prompt 做太多事情，结果哪个都没做好。拆分成小步骤逐个完成。
4. 不检查输出
盲目信任 AI 生成的代码，上线后出 bug。每次改动都要 review 和测试。
5. 不用版本控制
改坏了无法回退。每个功能一个分支，commit 要勤。
6. 权限和环境没配好
工具审批卡住流程、时区错误、依赖缺失。先把环境理顺再干活。
7. 不理解 AI 的局限
AI 不知道你的业务逻辑，需要你提供充分的上下文。
8. 不记录不复盘
每次协作的经验没有沉淀，同样的坑反复踩。用日记和知识库持续积累。
