## 核心原则

**做事三层次（做正确的事 → 正确的做事 → 把事情做正确）**

源于 Peter Drucker 管理哲学：

| 层次 | 含义 | 实践 |
|------|------|------|
| 做正确的事 | 先确定目标/方向是对的 | 探索→计划，理解问题本质再动手 |
| 正确的做事 | 用正确的方法/流程 | Plan Mode、验证方式、模块化 |
| 把事情做正确 | 一次性做对，减少返工 | 测试驱动、视觉验证、/clear避免污染 |

**这是元原则，优先级高于所有具体技巧。违反它时所有技巧效率都降低。**

---

**遇到需要人干预的场景时：**
1. 首先考虑是否有自动化方案或命令可以实现
2. 如果自动化可行，**选择不需要我参与的方案**
3. 绝不因为"习惯了问用户"而放弃自动化探索

**Rule: 优先自动化验证，自动化可行时绝不让人介入**

**方案先行原则：**
1. 先分析依赖关系和风险点
2. 方案没想清楚之前不轻易动手
3. 把前置条件、依赖项、潜在风险都列清楚再执行

## 元原则框架（完整体系）

### 思维层 - 第一性原理

**来源：Addy Osmani、Engineering Management**

| 原则 | 含义 | 实践 |
|------|------|------|
| 分解问题 | 拆解到最基本组件，再重新组装 | 遇到复杂问题先拆解 |
| 问5次Why | 追问直到根因 | 调试时连续问5次为什么 |
| 挑战假设 | 区分市场/问题/方案/人的假设 | 不接受理所当然的设定 |

### 设计层 - 软件工程原则

**来源：KISS、DRY、YAGNI、SOLID**

| 原则 | 含义 | 实践 |
|------|------|------|
| KISS | Keep It Simple 优先简单解 | 任何复杂方案先考虑简单方案 |
| DRY | Don't Repeat Yourself | 避免重复，逻辑只在一处 |
| YAGNI | You Aren't Gonna Need It | 不做过度设计，只做当下需要的 |
| SOLID | 单一职责、开闭、里氏替换、接口隔离、依赖反转 | 面向对象设计准则 |

### 执行层 - AI Agent Loop

**来源：Claude Code Agent Loop、Think-Act-Observe-Correct**

| 原则 | 含义 | 实践 |
|------|------|------|
| Think→Act→Observe→Correct | 核心循环 | 每个行动后观察结果再纠正 |
| 自纠正 | 默认持续直到解决，非一次就停 | 不要做一次就停下 |
| 有界但开放 | 上下文限制 + 用户控制 + 自然终止 | 设置 max_turns 等边界 |

### 行为层 - 记忆管理

**来源：Claude Code Agent Loop**

| 原则 | 含义 | 实践 |
|------|------|------|
| 上下文保护 | 上下文压缩时保留关键状态（文件修改、待办） | /clear 时确保重要状态不丢失 |
| 定期清理 | 无关任务清理，保持上下文干净 | `/clear` 清理无关对话 |

### 修正层 - 调试原则

**来源：systematic-debugging skill**

| 原则 | 含义 | 实践 |
|------|------|------|
| 根因优先 | 不在症状处修复，追溯到原始触发点 | 修复前先找到根因 |
| 3次失败停止 | 连续失败说明是架构问题，非战术问题 | 连续3次失败时停下来重新分析 |
| 系统化方法 | 系统化方法95%首修率 vs 临时应对40% | 用方法论而非碰运气 |
| 红牌警告 | "先凑合修复"、无假设就尝试、不理解就假设 | 这些情况出现立即停止 |

## Claude Code 官方最佳实践

### 给 Claude 验证方式（最高杠杆原则）

> **这是单件事中杠杆最高的一件。**

Claude 在能验证自己工作时表现惊人得好——运行测试、对比截图、验证输出。

| 策略 | Before | After |
|------|--------|-------|
| **提供验证标准** | "实现邮箱验证函数" | "实现 validateEmail，测试用例：user@example.com 是 true，invalid 是 false。写完运行测试" |
| **视觉验证** | "让仪表盘更好看" | "实现设计，截图对比原图，列出差异并修复" |
| **解决根因** | "构建失败了" | "构建失败报这个错：xxx。修复并验证构建成功，解决根因而非压制错误" |

### 立即纠正

- 发现 Claude 偏离轨道时立即纠正
- 反复纠正超过2次 → 说明上下文已经污染
- **修复方式**：`/clear` 后重写更好的初始 prompt

### 探索 → 计划 → 编码

1. **探索**（Plan Mode）- 读文件理解问题
2. **计划** - 创建详细实现计划
3. **实现** - 切换 Normal Mode 编码
4. **提交** - 创建 PR

## n8n 调试最佳实践

### 自动化验证优先
- 用 `mcp__n8n__get_execution` API 查询结果，不问用户
- 检查 `status`, `lastNodeExecuted`, `runData`

### 模块化设计
- 拆分为小 workflow，用 Execute Workflow node 调用
- 单个组件独立测试，出错时快速定位

### 错误触发器
- Error Trigger 节点，失败自动通知
- Retry on Fail 处理临时错误

### 输入验证前置
- 流程早期 IF 节点验证数据格式
- 防止后续难以追踪的错误

### 测试隔离
- 复制生产 workflow 用测试数据验证
- 不在生产环境直接调试

## n8n 技术要点

- `mcp__n8n__get_execution` 查询执行结果
- HTTP Request 替换输出，用 `$("NodeName").item.json.field` 访问之前数据
- `bodyRaw` SDK 参数名是 `body`
- 嵌套 `jsonBody` 表达式不解析，用 `body: '={{ JSON.stringify(...) }}'`

## 主动记忆规则

**技术调试/解决方案自动记忆：**
- 遇到技术问题并找到解决方案时，**主动**保存为 feedback 记忆
- 保存位置：`/home/ubuntu/.claude/projects/-home-ubuntu/memory/feedback_*.md`
- 不需要用户提示"记一下"，自行判断并保存
- 内容包括：问题现象、根因分析、解决方案、关键代码/配置

**记忆触发条件：**
1. 调试超过 2-3 个步骤才找到根因
2. 某个错误是首次遇到
3. 解决方案有参考价值（不是显而易见的）
4. 技术栈关键经验（如 API 格式问题、端口占用、权限等）

**日记 vs 记忆：**
- 日记 = 用户生活/工作记录，**不主动生成**
- 记忆 = 技术经验教训，**主动保存**
